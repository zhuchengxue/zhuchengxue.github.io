# 学语思日常运维

## 平时只做两件事

第一，用 Obsidian 或 Typora 写文章，并把写完的 Markdown 放进 `02-待发布/`。第二，双击“发布文章”，选择文章并点“发布到博客”。

发布器会依次完成：获取 GitHub 最新版本、读取 Dropbox 原稿、整理图片、提交并推送、触发 GitHub Pages 构建。成功后程序自动退出。

博客发布完成后，原稿会自动移入 Dropbox 的 `03-已发布/`。博客失败时原稿留在 `02-待发布/`，修复后选择同一篇文章重试即可。

## Dropbox 写作桌面

发布器会自动维护这套结构：

```text
公众号文章/
├─ 00-今日写作.md
├─ 01-选题池.md
├─ 02-待发布/
├─ 03-已发布/
├─ 04-素材/
└─ 05-归档/
```

根目录只放草稿入口和选题，不再放待发布文章。真正要上线的文章统一放进 `02-待发布/`。发布器只负责发布，不负责打开编辑器。

## 文章如何判断是更新还是新建

首次发布时，发布器会把 Dropbox 文件名记录为文章的稳定身份。之后可以修改文章标题，也可以在 `03-已发布/` 里继续编辑，博客仍会更新原文并保留网址和发布日期。不要在发布后随意重命名 Markdown 文件；确实需要改文件名时，可请 Codex 协助迁移身份。

## 公众号草稿

公众号自动草稿暂时停用，不进入博客发布流程。以后具备固定出口 IP 时可以重新启用；现阶段公众号内容请在后台手动处理。

## 常见问题

### 博客没有立刻更新

GitHub Pages 通常需要一两分钟。到 [GitHub Actions](https://github.com/zhuchengxue/zhuchengxue.github.io/actions) 查看最新任务；绿色表示发布完成。

### GitHub 推送失败

先打开 GitHub Desktop，确认已登录并能看到该仓库，再重新同步。若另一台电脑刚发布过文章，发布器会自动拉取并重试；只有两台电脑同时修改同一篇文章时才会停下来保护内容。

### Dropbox 文章找不到

确认 Dropbox 已完成同步，并在 Obsidian 中打开正确的“公众号文章”Vault。新文章要放进 `02-待发布/`，根目录的 `00-今日写作.md` 和 `01-选题池.md` 不会出现在发布列表里。也可以在“本机设置”中直接填写写作库路径。

### 图片找不到

图片应位于文章同目录、Vault 根目录、`attachments/` 或 `附件/`。Obsidian Wiki 图片和普通 Markdown 图片都支持。

## 高级维护

完整本地验证可运行 `npm test` 和 `npm run build`。这些只用于开发或排障，不属于日常写作流程。
