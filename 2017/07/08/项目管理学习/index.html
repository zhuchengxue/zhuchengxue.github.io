项目管理学习

发表于 2017-07-08 | | 本文总阅读量 次

一、企业痼疾

一开始先列举了中小企业普遍存在的一些“痼疾”现象，比如“老大难”、“公司不大，皮球不小”、“部门墙”，

接着指出发生这些现象的问题所在：

  1. 人才短缺
  2. 资金紧张
  3. 管理不规范
  4. 质量不稳定
  5. 有单做不出来
  6. 设备经常坏
  7. 员工能力不足
  8. 人员难以管理

一个公司或多或少都会有这些问题。
抛出了问题，那就需要解决问题，如何解决呢，感觉和上次差不多，殊途同归，也是以项目为中心的管理方案，这里是叫做集全厂之力—项目化管理 ，接着引出介绍了一些项目管理的知识，

二、什么是“全面项目化管理”？

背景简介

举例华为开始倾斜以项目为中心运作，具体来说：项目组是华为的最小组成部分，部门负责人是部长，项目组负责人是PL（project leader），三个项目组还有一个PM（project manager）

发展过程

项目管理发展过程，可以分成了四个层次：

  • level 1. 事件管理 这种管理随机性大，主次轻重分不清，属于最低级的管理
  • level 2. 重点事件管理 比level 1稍微好点，开始注重重点事件；
  • level 3. 项目化管理 这种管理特点优化重点事件的质量和流程，可以科学化、模块化的管理重点事件；
  • level 4. 全面项目化管理 不分轻重，统一按项目管理，以目的为导向，有体到面进行管理。

现在

项目管理已成为企业管理最常用的工具，与MBA齐名，有专门的认证，并且有专门的软件管理，运用项目管理最成功的三个行业：软件开发、建筑、投资

全面项目化管理内容

主要分为两类：

项目类工作：按项目管理，具有临时性和独特性特点

例行工作： 先分类打包，再做项目管理

所谓的全面化项目管理是指：公司各部门依据公司发展战略，制定出适用于各部门的发展策略，然后制定出相应的年度工作计划，再根据计划，对工作进行分类，制定出各部门的项目清单，最后统一由项目管理中心来管理项目。

全面项目化管理好处：

  • 部门工作分类更准确，条例清晰，职责分工更明确，更利于摆脱混乱的局面
  • 按项目方法管理更容易监管，责权利一致，可实现目标管理和量化管理
  • 更有利于工作交接和人才培养、知识管理和沉淀
  • 全面项目化管理形成了矩阵式的管理机构，彻底打通公司上下及部门间的管理。
  • 承接战略目标，降维计划内容，将重点工作一网打尽，项目抓好了，管理自然就轻松了

那个问题的其中一个答案

两个人比如甲和乙

  1. 甲拖地30min，同时乙先擦窗15min，15min切菜（此时拖地完成，擦窗和切菜还各需要15min才能完成）
  2. 甲擦窗，乙切菜，同时做15min，完工

三、全面项目化管理流程

启动阶段——计划阶段——实施阶段——收尾阶段

监管贯穿始终

对各阶段如何进行项目化管理

① 启动阶段

注意事项

  • 识别项目的需求
  • 确定项目的目标
  • 定义项目干系人的期望值
  • 描述基本的项目范围
  • 选择基本的项目组成员

开展工作

  1. 立项申请
  2. 组建项目组
  3. 项目策划/制作任务书
  4. 项目开工会

输出

  1. 项目组成员表
  2. 策划报告、任命书

启动阶段管理要点

项目人员

  1. 与客户、领导充分沟通，明确需求、获得支持
  2. 明确项目目目标和定位
  3. 统一项目成员思想、明确相关游戏规则
  4. 安排好工作、预留项目时间

公司管理者

  1. 选择好项目负责人，项目就成功了一半（即解决问题，也可以培训）
  2. 平衡好下属的工作及时间
  3. 约定好激励措施
  4. 为项目打气、协调资源、坚定支持。

② 计划阶段

注意事项

  • 在质量、时间和资源中取得平衡，从而进一步明确范围
  • 活动清单和次序
  • 进度表
  • 预算
  • 风险分析
  • 管理计划（成本、时间、范围、风险管理计划等）
  • 得到相应的项目干系人批准的项目计划

开展工作

  1. 工作分解
  2. 活动排序
  3. 资源、工期、成本估算
  4. 风险计划、沟通计划
  5. 项目计划

输出

  1. 工作分解表
  2. 网络图/甘特图
  3. 进度计划
  4. 风险计划
  5. 沟通计划

项目计划制定流程

主要的顺序：

  1. 工作分解
  2. 活动顺序
  3. 资源工期成本估算
  4. 进度计划 风险计划 沟通计划
  5. 项目实施
大事化小

项目要做的事情一般都很多、很杂，一下想不清楚，所以才要分解。
分解的方法：自上而下，头脑风暴
表达方法：图形式，目录式
将项目的任务按照逻辑进行底层细分，分解到可预测、可管理的最底层单个活动为止。

项目任务分解成功的标识：

完全穷尽，彼此独立

  • 一个清晰的任务完成
  • 一个清晰的责任人
  • 能够估算工作量和工期
  • 通常工作不超过两周
工期估算

定义

根据项目范围和资源的相关信息，确定（估计）完成所有活动所需的工时。
常用估算方法：

  1. 三点估算：

公司 Te=（To+4Tm+Tp)/6

  • To: 最乐观时间，为基于活动的最好情况，所得到的活动持续时间
  • Tm: 最有可能时间，为基于活动最有可能活动持续时间
  • Tp：最悲观时间，为基于活动的最差情况，所得到的活动持续时间
  • Te：预期活动持续时间
  1. 专家判断 专家判断利用过去相似活动经验的专家意见来估算工期
  2. 类比估算 使用以前实际项目中完成的类似活动所需要的时间来估计当前活动的时间，以前的活动有相关的文档
  3. 根据工作量估算 由工程或设计部门确定的每项具体工作种类所需要完成的数量，乘上单位生产率。
需要思考的问题

我要完成这些任务：需要什么资源？什么时候需要？需要多少？我要的资源在哪里？由谁批准？

关键路径上的任务，绝对不能延迟

计划阶段管理要点

项目人员

  1. 不能做独狼，要团队合作
  2. 工作任务要充分分解
  3. 风险要提前识别并有针对性的防范措施
  4. 要根据项目需要，提前做好沟通和协调

公司管理者

  1. 特别要注意审核风险点的识别
  2. 对于工期和关键路径要重点关注
  3. 要特别留意里程碑节点，是否在沟通计划中体现
  4. 项目目标是否合理？工期是否达到要求
  5. 项目所需资源是否满足？

③ 实施阶段

注意事项

  • 保证执行计划所需要的资源
  • 领导团队
  • 和项目成员开发
  • 监控项目进展，衡量绩效及与干系人沟通
  • 识别并解决问题
  • 化解冲突

开展工作

  1. 沟通
  2. 项目监控
  3. 变更管理

输出

  1. 项目会议记录
  2. 项目状态报告
  3. 项目变更申请

④监控阶段

注意事项

  • 识别计划的偏离
  • 采取矫正措施以使实际进展与计划保持一致
  • 接受和评估来自项目干系人的项目变更请求
  • 必要时重新调整项目活动
  • 必要时调整资源水平
  • 得到授权者批准后，变更项目范围
  • 调整项目目标并获得项目干系人的许可

方法与工具

  • 进度计划表
  • 里程碑
  • 会议
  • 稽查
  • 项目状态报告
  • 评估报告

项目人员

  1. 严格进度计划进行
  2. 要确保沟通充分
  3. 高风险，关键路径的工作，要确保。
  4. 如实记录好团队成员的表现
  5. 会议记录，状态报告，变更管理要及时填写

公司管理者

  1. 特别要注意风险控制
  2. 对“里程碑”和”关键路径“的工作要盯紧
  3. 再忙，也要抽时间关注项目进度
  4. 项目目标的实时状态
  5. 项目变更的合理性和充分性
  6. 最好有专人或专门组织跟进项目进度
  7. 项目的完成情况要与绩效、奖金、晋升……等挂钩
  8. 项目人员的表现和工作能力的提升

⑤ 收尾阶段

注意事项

  • 承认功绩和成果
  • 得到最终项目的干系人的认可，合同终结
  • 项目经验教训的最终定稿
  • 和包含项目组成员在内的项目干系人一起评估项目过程/成果
  • 撰写项目总结报告
  • 关闭项目办公室，解散项目组

收尾阶段管理要点

项目人员

  1. 提前准备项目的全套文档、资料
  2. 与相关部门一起评估目标达成情况
  3. 及时与团队进行总结得失、作知识沉淀
  4. 识别关联的下一步行动计划
  5. 激励与感谢

公司管理者

  1. 重点评估目标达成，计划与实际的差异
  2. 重承诺，按原定的激励措施实施。建立要及时
  3. 再忙，也要抽时间参加项目总结会
  4. 关注文档/资料的保存，知识沉淀
  5. 下一步的行动计划
  6. 表扬优秀，指出不足

四、 项目成功的因素

  1. 项目目标和目的、工作范围是否明确
  2. 是否获得领导的积极支持
  3. 项目成员积极性、工作能力是否足够
  4. 沟通是否有效
  5. 目标时候合理
  6. 变更控制是否严格
  7. 是否约定好相关的游戏规则
  8. 成员工作能力、态度和项目管理知识是否足够
  9. 是否建立了良好的，积极的，团队合作工作氛围
  10. 项目负责人经验是否丰富

几点忠告

  1. 任何一家公司，都不可能靠一两个工具就可以管理成功
  2. 在中小企业实行项目化管理，必须有其它的配套措施
  3. 推行过程中，由易到难，由局部到全面，切忌遍地开花
  4. 推行前期，领导人必须全程跟进，尽量听取进度汇报
  5. 在中小企业，项目跟进人员必不可少
  6. 所有的项目变更必须审批（借此总结经验）

五、项目失败的主要原因

  1. 员工未把完成项目当成必须承担的义务
  2. 项目工作没有方向
  3. 没有激励措施，或责权利不一致，员工没有动力
  4. 公司不重视，领导不跟进，没有后续配套制度
  5. 没有明确的目标、目的，或目标过高
  6. 公司为提供有效的支持货资源
  7. 不断变化的要求，项目变更太多
  8. 沟通失败
  9. 项目负责人能力欠缺
  10. 企业文化/工作氛围欠佳

六、项目管理需要用到的表单

  1. 项目组成员
  2. 策划任务书
  3. WBS表
  4. 项目进度计划表
  5. 风险管理分析表
  6. 沟通计划表
  7. 会议记录表
  8. 状态报告表
  9. 项目变更管理表
  10. 项目总结表

七、IVD体外诊断试剂研发各阶段

基于市场驱动的研发，简单来说，就是一切向着能赚钱的方向努力，这个项目务必是有巨大的前景的并且进展迅速的。

现在我们按照项目进度所划分的文件，如项目过程文件、技术文件、注册文件三部分，其实并不是隔离分开的。相反，这三类文件是互相嵌套的。具体来看，项目过程文件包含注册文件、技术文件，而所有的文件只要涉及到相关的记录都需要配套补齐。

通过前期对于项目管理方面的学习，现在细细总结这个研发体系到底应该怎么展开，按照里程碑划分，主要包括

  • 立项阶段
  • 研发阶段
  • 转产阶段
  • 临床确认阶段
    注：每一个阶段都需要做评审

立项阶段

立项过程，主要解决的问题是在于项目的可行性的分析，其中应该包括技术、成本以及时间控制可行性分析，制定好计划和方案，需要输出的文件：

  1. 立项建议书
  2. 综述资料初稿
  3. 项目的可行性分析报告
  4. 项目任务书
  5. 设计开发计划
  6. 设计开发方案
  7. 风险管理计划
  8. 风险管理分析
  9. 综述资料初稿

研发阶段

研发阶段，主要解决的问题在于验证按照计划拟定的内容制定的方案的可行性，从而定出这个项目的技术要求，并对关键技术和生产工艺条件进行确认，需要做小试，需要输出的文件：

  1. 主要原材料研究资料
  2. 参考品或标准品的溯源资料
  3. 主要反应体系及生产工艺研究资料
  4. 稳定性研究资料
  5. 阳性判断值或参考值范围研究资料
  6. 分析性能评估资料
  7. 产品技术要求
  8. 产品说明书
  9. 其它的技术文件（生产SOP、质检检验标准、BOM等）

转产阶段

需要采购、生产、质检、仓库配合，完成中试产品的生产制造，前期需要完成这个项目相关验证，需要输出的文件：

  1. 设计开发验证计划
  2. 设计开发验证方案
  3. 设计开发验证报告
  4. 试产总结报告
  5. 试生产申请书
  6. 试生产总结报告
  7. 风险管理报告

送检与临床

  1. 临床实验方案
  2. 临床合作协议
  3. 临床实验报告
    这个是由公司其他部门来完成，研发主要的工作在于配合。

八、最后一点感想

  • 众人拾柴火焰高，最靠谱的是聚焦所有人的注意力到要做的项目上去；
  • 管理的方法论是非常重要，但做项目的人更重要，做项目的项目负责人最重要；
  • 比如有些人你看到都在努力工作，但转身后马上淘宝手游走起，还有些人也努力工作，也不玩淘宝，不玩游戏，看过去兢兢业业，但绩效一般；前面一种叫出工不出力；后一种叫出力不出活。尽力来自本分，尽心来自善意 ，你从来不可能真正激发一个人，你只能给他一个理由，让他激发自己。流程化管理是规定了每月每周每天都该做什么，但真正做事的时候的态度，做实验按部就班能够解决问题？这些方法论是非常好，但如何激发人“想干”或许是应该更需要考虑的问题
  • 现在公司或者整个部门的问题，并不是不知道要做什么事情，做这些事情也不是没有方法论，而是做这些事情有没有盼头,进展的速度是不是够快，一直停滞不前总会耗光所有的热情；
  • 管理的方法论，比如上次的“结构化流程”，这次的“全面项目化管理”，所谓的结构化流程，不就是不断的分解任务而已，而全面项目化管理，就是把这些分解好的任务按照所谓的“项目”来工作。到最后还是没太大差别没有，监管分配任务，员工例行公事。
  • 作为管理者，应该尽量设计好一个周密的系统，解决如何有效激励，说白了如何分钱，我们公司有一套这个系统，但是下面员工并不知道做完这个项目到底可以拿多少钱，暗箱操作给人的感觉领导说了算。
  • 没有功劳也有苦劳，功劳需要付钱，苦劳需要鼓掌，分辨功劳与苦劳，而不是纯粹的鼓励多加班，加班就能够多做事？研发又不是生产。
  • 这么多的新型管理方法一定是好的吗？管理从来应该来自实践，有用的才是好的，当然借鉴是对的。
  • 要知道每一个人都是不同的，知人善用从来都是不容易的，把合适的人放在合适的位置上。
  • 最终的绩效，对我们研发部门来说出新产品才是目的，而所谓的管理都是要以这个为目标为导向。

整理了好久，一点碎碎念，还有很多需要完善，一份初稿。
